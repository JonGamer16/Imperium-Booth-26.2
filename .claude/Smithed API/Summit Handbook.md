# The Build of your Booth

The physical build of your booth can be considered as two parts: the blocks that compose your structure and the entities you fill inside of it. We highly encourage you to leverage the [display entities](https://minecraft.wiki/w/Display) in building your booth as they make it very easy to showcase custom items and add text to walls, etc. to be able to showcase and talk about your project.

![][image12]  
*Examples of text displays being used in Summit ‘24*

## Static vs Dynamic Entities {#static-vs-dynamic-entities}

These sorts of displays are considered **static entities** as they are unchanging throughout your booth’s experience. Since they are purely used for augmenting your build with extra displays and models, they can be considered a part of the build and should be submitted along with the schematic.

You may also have entities that you need to manipulate whether it’s an Animated Java rig, an interactive bookcase, or just some part of your booth that needs to be updated. Any entity that needs to be manipulated by a command in your data pack is considered a **dynamic entity**.

This distinction ends up being important for the production Summit server as we will run a custom fabric mod that avoids ticking entities marked with `summit.static` at all. We will manage this for you by auto-tagging entities submitted in your schematic (more information about this in the [**schematic section**]().)

# Designing Booth Interactions 

While not required for every booth, we encourage most booths to leverage data packs to implement some form of interactivity with the guests. From implementing a custom item to share as a gift, to creating a custom experience or mini-game – there’s a lot of room for creativity in the designs for your booth.

It’s really important for a booth to function well in the server’s unique execution environment. This means you’ll need to adapt your booth to work smoothly by not interacting with global aspects of the world and not interfering with other booths’ interactions.

We’ve set up this portion of the docs via the subsection to dive into the various aspects of the BSL and how you should best leverage it for your booth’s data pack\!

# ⭐ Booth Definition

# **Booth Definition**

This file defines your booth’s metadata which includes information such as the name of your booth, the dimensions of the bounding box that defines it, and more. Each booth is **required** to define this file (and only one) which will be submitted alongside your data and resource packs. We’ve explained the exact structure of this file alongside which fields are required or not at the end of this page.

**No** fields should be namespaced with the exception of resource locations, such as your functions and textures. Every other field unique to your booth will use the supplied ‘namespace’ field.

***Note**: this definition is growing and changing as we add new systems so look out for updates in the vendor announcement channels in the discord\!*

## Field Explanations

- namespace :  For your submitted data pack and resource pack, you are only permitted to use one namespace, unless you have extenuating circumstances. List that namespace here. This acts as an id for your booth.  
- booth\_name :  The name of your booth, this is used for display so you can use characters like &.  
- bounding\_boxes :  This is an object containing named spacial bounding boxes for your booth. Think of these as arbitrary cuboid volumes that you can define. Combined they should cover the entire area of your booth.  
  - \<bounding\_box\_name\> :  Name the bounding box whatever you want  
    - position :  Defines the area of the bounding box itself, with each value being a float. See above.  
    - on\_player\_enter :  (Optional) A function location that will be run as and at a player when they enter this specific bounding box  
    - on\_player\_exit :  (Optional) A function location that will be run as and at a player when they enter this specific bounding box  
- on\_player\_enter :  (Optional) A function location that will be run as and at a player when they enter your booth as a whole. (When a player goes from not being in any bounding box to being in at least one)  
- on\_player\_exit :  (Optional) A function location that will be run as and at a player when they exit your booth as a whole. (When a player goes from being in at least one bounding box to not being in any)  
- persistent\_tick:  (Optional) A function location that will be run tickly on a persistent basis. **Ask your coordinator before using this.**  
- ticking\_functions :  (Optional) A map of function locations and their timings for ticking. These will only be run when a player is in one of the bounding boxes you defined above for your booth. If you think you have reason to be ticking functions otherwise, please ping your assigned coordinator to inquire.  
- sticker\_book :  (Optional) This summit we will be providing players with a sticker book. Booths will be able to register stickers through here and then grant them how they want via generated advancement.  
  - background\_color :  Each booth will have a strip of tape where their stickers are displayed in the sticker book. Define the color of your tape here as a hex code.  
  - region :  Either patched\_plateaus, textured\_tropics, or welded\_woodlands.  
  - stickers :  Each booth is permitted up to 5 stickers. List them here.  
    - sticker\_id :  A technical id for this sticker, these are automatically namespaced using your booth’s namespace to avoid collisions with other booths. The advancement for the sticker will be located at summit.sticker\_book:\<namespace\>/\<sticker\_id\>. The advancement is generated and should be granted to reward the player with the sticker.  
    - sticker\_texture :  A resource location for a texture in your resource pack to use for the sticker, e.g. my\_booth:sticker/on\_enter  
    - sticker\_name :  The name of this sticker, will show on hover.  
    - unobtained\_text :  Text to display on hover for this sticker when it is yet to be obtained. When unobtained, stickers will show as a silhouette of the sticker.  
    - obtained\_text :  Text to display on hover for this sticker when it has been obtained.  
- balloons :  (Optional) Balloons are a cosmetic item that booths can give out to players. Booths will be able to register balloons through here and then grant them how they want via a generated function, or via the balloon vendor as described in the Interaction API section.  
  - balloon\_id : A technical id for this balloon, these are automatically namespaced using your booth’s namespace to avoid collisions with other booths. A function for the balloon will be located at summit.balloon:give/\<namespace\>/\<balloon\_id\>.  
  - balloon\_name : The name of this balloon, will be the name of the balloon item.  
  - balloon\_model : A resource location for a model in your resource pack to use for the balloon, e.g. my\_booth:balloon/basic

# Pasting your Booth

After onboarding, you’ll be provided with a copy of the schematic alongside a tp command with exact coords and some brief instructions ([**example**](#booth-selection)). Most importantly, you should ensure the following things are true:

- /gamerule random\_tick\_speed 0  
- //perf off 		(World edit command)

Be very careful with block updates. Ideally use axiom with the No Updates option enabled.

# Schematic Entity Transfer

When you submit your booth, you will submit a WorldEdit schematic with any [**static entities**]() you want to include. (Make sure to use \-e to the copy command)  
These entities will be stripped out of the schematic and automatically moved to summon commands.

However, transferring entities is not simple and there are a lot of caveats; this will only transfer the data on the entity, **not scores**.

Therefore, any entities in your booth that use scores (including Animated Java rigs) should be summoned and killed via functions and **should not** be included in your schematic.

There are 2 function tags that you should add these summon and kill commands of these special entities:

- \#summit.booth:\<namespace\>/entities/summon  
- \#summit.booth:\<namespace\>/entities/kill 

Where  \<namespace\> is the same namespace you specified in your [**Booth Definition**]().

You can of course use these function tags for your static entities as well, but keep in mind, if you handle entities through these summon and kill function tags, DO NOT LEAVE THEM IN YOUR SCHEMATIC.

# Saving your Booth

First, make sure your booth is in the exact starting scenario that you wish. This will always represent the start state. You should save your booth using worldedit in the **exact** same spot that you pasted it in. You can find the exact coordinates by scrolling up in your booth thread.

The commands for saving a WorldEdit Schematic are as follows:  
//copy or //copy \-e  if you need entities.  
//schematic save \<name\>

If you wish to ensure certain blocks stay interactable, you can add special marker entities that will enable specific block interactivity using the [**summit.enable\_block\_interactivity**]() tag. This will bypass our custom worldguard system.

# ⭐ Ticking your Booth

# **Ticking your Booth**

For Summit, we completely handle all of the ticking logic for your booth. In order to register a ticking function, you’ll define a new entry in the ticking\_functions field of your [**Booth Definition**]() letting you define specific functions to be ticked at specific rates (once a tick, once a second, etc.) This lets us only tick your booth if there are players within it and also lets us multi-thread all of the booths’ ticking code as we please.

# Booth Events

In order to detect when someone enters and exits your booth, you should refer to the bounding\_boxes field of your [**Booth Definition**](). This lets you define multiple bounding boxes that mostly encompass the space your booth needs for its interactivity. You can define functions for on\_player\_enter and on\_player\_exit to detect when a player enters and exits the entire booth as a whole alongside defining said fields for each bounding box within. This should give you a degree of control if you need special sub-booth regional events while still managing a player within your booth.

# Other Useful Techniques

In order to tick your booth, we generate a location predicate that checks for a player within the booth region. You can reference this predicate in any advancements as you please, more information [**here**](). We also tag the player with summit.in\_booth.\<namespace\> which you can also leverage to detect if someone is in the booth, more information defined [**here**]().

# Persistent Booth Ticking

While we have designed Summit to have the majority of booth vendors leverage the booth event system for ticking their booth, there may come a moment where you require ticking commands outside of your booth’s presence. These should be considered carefully and with intention as to not add a ton of load to our servers.

## Schedule Clocks

These can be used to have controllable clocks at varying speeds for a temporary ticking sequence. For example, if you have an animated entrance that triggers when someone enters your booth, you may want to schedule a slow clock (like every 2t or 5t) that checks if the mouth needs to be closed if the animation state gets stuck and your booth stops ticking. You can use schedule function \<my\_func\> 5t replace  in a function that schedules itself until a condition is met to handle these sorts of cases. These should be used for short, burst ticking clocks and not for extended ticking sequences.

## Persistent Booth Ticking

If you have a function that needs to be ticked every tick (even within some condition), there will be a persistent tick available for your booth that will be triggered every tick. You should *only* leverage this in the case where you have some heavy or expensive logic (mostly things that require commands every tick) that needs to work outside of someone within your booth region. These will be closely monitored and approved case by case. You should always have a single score or flagged condition that precedes any expensive selector or function call to try to reduce the number of tickly commands as much as possible\!

See the persistent\_tick field on the [**Booth Definition**]() page.

# ⭐ Entity Tags

# **Entity Tags**

summit.static

- Any entity tagged with summit.static will not send any updates to the client. You can still target and update them, but it will not present any changes visually as [**they will not tick**]().  
- By default, any entity included in the Schematic will be tagged with summit.static, but you can override this by tagging it with summit.dynamic instead.

summit.booth\_entity.\<namespace\>

- Ensure to tag **ALL** of your entities with this. We need to be able to target the entities exclusively from your booth. Entities included in the booth schematic will automatically be tagged with this.

summit.in\_booth.\<namespace\>

- If a player is within any one of the bounding boxes you defined in the [**Booth Definition**](), they will be tagged with this. Use this to target players in your booth.

summit.enable\_block\_interactivity

- Summon a marker entity on specific blocks you wish to keep its interactivity. The production summit server will leverage a worldguard type system to ensure booths are maintained and not disrupted / updated. If you wish to enable a block’s normal interactivity (such as GUI blocks, doors, levers, etc), you can spawn a marker entity with this tag and submit that as a part of your schematic.

# **The Sticker Book**

All players will receive a sticker book item that, when right clicked, will open up a custom book GUI. Booths can register stickers via the [**Booth Definition**](), with the intention that stickers will be given to players for doing things in the booth, such as visiting it, finding a secret, or participating in some kind of activity.

Booths can register up to 5 stickers, and they’ll all be placed into the sticker book together, with the booth name above it. Unobtained stickers will automatically show a grey silhouette of the sticker. See below for an example of a page spread populated with the stickers of 5 different booths:

# Style Guide

Stickers are relatively flexible in terms of size and style, but we do have some guidelines:

* Stickers must use a square canvas (eg: 100x100).  
* Stickers should be at least 20x20. This is the resolution they will be scaled to in the sticker book, if you want to make your stickers pixel consistent, use this resolution. Stickers larger than 256x256 will be scaled down to 256x256.  
* Try to make your stickers be as large as possible on their canvas, otherwise they may appear somewhat small in the sticker book.  
* Alpha values will be preserved.  
* Don’t give your stickers drop shadows, they end up looking like they’re floating when placed in the sticker book.  
* Try to make your stickers look like stickers, giving them a white border can go a long way.

# Extra Info

* The score summit.total\_unlocked\_stickers stores the amount of stickers the player has unlocked. This score is updated whenever the player unlocks a sticker and recalculated whenever the player joins the server, meaning it persists over server shards.

# Droppable

By default, players will not be able to drop any items during Summit. There will be trash cans provided given your wish to dump unwanted items into them. If your booth's items desire to be droppable, you can place summit: {droppable: {}} in your custom\_data component to allow for your items to be dropped in certain ways:

kill  
If you would like the item to be killed when dropped, set droppable: {kill: true}.

callback  
If you would like to run a function on the dropped item, add droppable: {callback: true}. Then add your function to the \#summit.items:droppable/callback function tag and filter @s based on your item's data.

execute if items entity @s contents stone\[custom\_data\~{my\_cool\_item: true}\] run …

ignore  
If you want us to ignore your item completely, just add droppable: {ignore: true} to your item. 

This is equivalent to having a callback w/ no behavior (so it’s more “optimized”).

# Untrashable

Unless you have a very good reason, you should make sure all of your items are trashable. If you need an exception, you can ask us and we can allow you to add {summit: {untrashable: true}} to your item to prevent trashing.

# Persistence

By default, all items that are given out in your booth will be deleted the moment a player steps outside. This is to ensure that a booth's experience doesn't interrupt other booth's experiences. However, we still would like to encourage booths to give out *persisted*  items. To give an item that will **not** be deleted, add {summit: {persist: {}}} to your item.

There will be some extra rules you will have to follow in order for your items to be within our standards.

\- Your item **cannot** have the droppable tag  
\- Your item **cannot** be majorly invasive in any way  
  \- No block breaking or placing  
  \- No excess particles and sounds  
  \- Little to no ticking logic  
  \- No crazy attributes  
     \- Cannot change attributes more than 2x vanilla (and purely to our discretions)

suppress

The items you give out may have some set of interactions with the world. These might be a custom right click action that blows bubbles, gives you jump boost, or plays a sound. You may also give out items with slight attribute changes. These are fine to give out but you are **required** to add the suppress tag.

Adding {summit: {persist: {suppress: true}}} to your item will enable our [**Item Suppression API**]() to temporarily block your item's presence in the inventory (such as attributes, custom abilities, etc.) Certain booths will be able to control when your inventory should and shouldn't be active so that custom items don't interrupt a booth's experience.

Item suppression involves storing your item inside of a special container-like item. This item will have a custom “disabled” texture and name alongside a custom container component which will hold your item. Booths will be able to call a function to suppress and re-enable your items. If you leave a booth or the server, your items will be re-enabled when you join.

If you wish to disable these items for your own booth, please see the [**Item Suppression API**]().

# Item UUIDs

You may find yourself in a situation where you want items to not stack despite having the same data. This can be mitigated by setting a random value acting as an ID in the item’s custom data. 

We have created a system that will make this easy for you. Simply call the summit.items:api/generate\_uuid function to generate a new item UUID and then apply it via the summit.items:api/set\_uuid item modifier.

Example usage :

```
function summit.items:api/generate_uuid
item modify entity @s weapon.mainhand summit.items:api/set_uuid
```

Best practice is to generate the UUID before applying it, rather than generating it after. This ensures that your item will have a brand new UUID.

# ⭐ Inventory Saving

# **Inventory Saving**

This API helps in saving and restoring the players inventory even in the case if the player logs out in one server and logs back in on another. This is due to the special usage of the [**Shard Syncing API**](#shard-syncing) which lets you define a custom PlayerDB like storage unit to be synced across our server shards.

In general, this API should be avoided in favor of the lighter touch of the [**Item Suppression API**](). Only reach for these functions if you are designing an experience that requires total control over the player inventory for things such as the [**Battlegrounds**](#battlegrounds-api) or a custom Boss fight. Please let your coordinator know if you are planning on using this API.

## summit.inventory\_mananagement:save

Call this function to save the player’s inventory to storage and clear ALL items. This will persist between shards and will automatically load back onto the player when you leave the server.

## summit.inventory\_mananagement:load

Call this function to load the player’s inventory. If you call this, it won’t be called again if you leave and rejoin the server.

# ⭐ Item Suppression

# **Item Suppression**

This API lets you suppress potentially disruptive items from interacting with the world by placing it within a temporary barrier item with a bundle. It can then be un-suppressed by taking the item back. It will only target items listed as summit.persist.suppress: true as per the [**custom item definition**]().

***Note**: The current texture and item for the “suppressed item” is in flux and will likely be changed with a custom texture and updated name and lore.*

## summit.suppress:apply

Call this function to suppress items that need suppression. This will ensure these items get un-suppressed either by an explicit remove call OR automatically when you leave the server / booth via the reset function.

## summit.suppress:remove

Function that removes any item suppression if active.

# Style and sizing

![][image35]

And we ask of you not to make the balloon any bigger than 1.5x1.5x1.5 blocks maximum. In the attached picture you can see some examples of rope lengths per size:   
8x8x8 pixel balloon=24 pixel (1.5 blocks) rope  
12x12x12 pixel balloon=32 pixel (2 blocks) rope  
18x18x18 pixel balloon=40 pixel (2.5 blocks) rope  
20x20x20 pixel balloon=48 pixel(3 blocks) rope  
These are guidelines, you're welcome to make balloons in whatever shape you want, as long as you keep the size comparable to the listed size categories.

# ⭐ Shader Dev Kit

# **Shader Development Kit (SDK)**

The SDK takes the form of a python script which will manage for you the injection of your code into the (vanilla) vanilla shader, and will merge multiple workspaces together.

***Important Note**: Making vanilla shaders work for multiple developers at the same time is tricky, and requires a very strict discipline. **Please read the conventions very carefully**.*  
*A shader which doesn’t respect them won’t be accepted.*  
*If you have **any** questions, please DO NOT WAIT and feel free to ping @neylz on discord.*

# Commands

You can grab the script from [here](https://gist.github.com/neylz/b4bebc5fc31c2ad88363d1431dc03242). Those are command arguments, so you would run them from the root of the resource pack:

```shell
python scripts/sdk.py <command> [options]
```

### build

Scans your setup'd shaders following the established convention, and will generate the necessary functions to inject your shader into the vanilla shader.

Does **NOT** requires to be ran at each file edit, only when:  
\- creating the initial setup for your local project  
\- adding a new file  
\- adding a new namespace  
\- changing a [*sdk.toml*](#sdk.toml-configuration) configuration

### patch

Downloads vanilla shader sources for a given Minecraft version and applies the SDK mixins and injections into the vanilla core shaders.

**Those files must be kept untouched once generated.** To modify a shader refer to the [conventions](#conventions) below.

Additional options:

* *\--version* / *\-v* (default: *26.1.2*) \- Minecraft version to fetch.

# Conventions {#conventions}

## File structure

This is a full example of how the file structure should look like.

```
assets/
├── <namespace>/
│   └── shaders/
│       └── include/
│           └── sdk/
│               ├── sdk.toml
│               ├── item.vsh/
│               │   ├── include.glsl
│               │   └── main.glsl
│               └── item.fsh/
│                   ├── include.glsl
│                   └── main.glsl
├── minecraft/ [do NOT edit]
│   └── ...
└── sdk/ [do NOT edit]
    └── ...
```

## sdk.toml configuration {#sdk.toml-configuration}

*sdk.toml* **must** exist at *assets/\<namespace\>/shaders/include/sdk/sdk.toml* and **must** contain an *id* vector of four 8 bits unsigned integers (0–255) that identifies this namespace at runtime.

Example:

```
# make sure to use a different one!!!
id = [10, 11, 12, 13]
```

## Shader scripting

Writing shaders with the SDK is separated into two main parts: the [dependencies source](#dependencies-injection) and the [main injection](#main-injection).

As an example, here is a rough illustration of the vanilla (after patch) *core/item.vsh*.

```c
#version 330

#moj_import <...>

in ...;
uniform ...;
out ...;

// --- SDK dependencies injection point here ---

void main() {
    // vanilla shader logic
    ...

    // --- SDK main injection point here ---
}
```

The dependencies injection and main injection points are mapped to your shader's *[include.glsl](#dependencies-injection)* and [*main.glsl*](#main-injection) files respectively.

### Main injection {#main-injection}

Path: *assets/\<namespace\>/shaders/include/sdk/\<pipeline\>.\<stage\>/main.glsl*

In this file, you write directly your own shader logic. The checks are hidden from you. You just write the function content, like if you were writing a normal shader.

A valid example:

```c
// assets/foo/shaders/include/sdk/item.vsh/main.glsl
vertexColor = vec4(1., 0., 0., 1.);
```

However, you can **NOT** define functions here:

```c
// assets/foo/shaders/include/sdk/item.vsh/main.glsl

void myHelperFunction() {   // not allowed!
                            // you are in the void main() function already!
    // some logic
}

vertexColor = vec4(1., 0., 0., 1.);
```

To overcome this, you can write that helper function in the [*include.glsl*](#dependencies-injection) file, and it will be available for you to call in *main.glsl*.

#### 		Vertex program specificities

In the vertex program, two additional variables (as read only) are provided.

| Type | Name | Description |
| ----- | ----- | ----- |
| ivec2 | sdk\_atlasSize | Sampler0 size in pixels |
| ivec2 | sdk\_iuv | UV of Sampler0 but premultiplied by sdk\_atlasSize |

### Dependencies injection {#dependencies-injection}

Path: *assets/\<namespace\>/shaders/include/sdk/\<pipeline\>.\<stage\>/include.glsl*

In this file, you can write any helper function or constant declaration. However, **you need to be careful**:

1. Too many constant definitions will bloat the shader, and we won't be able to accept your shader. Please prefer on the fly computations by a deterministic algorithm instead of hardcoding values.  
2. You can write functions here, but everyone can as well. So **it is absolutely mandatory** to [namespace](#namespacing) your functions.  
3. Adding additional inter-stage variables (ie. *in*/*out*) is **NOT** allowed. Checkout [sdk inter-stage](#sdk-inter-stage-communication) for workarounds.  
4. Moj imports can be used, under the condition of wrapping the imported file into an [include guard](#include-guard) to avoid conflicts.

### Namespacing {#namespacing}

To avoid conflicts, any constants, precompiler definitions, and functions you write in [include.glsl](#dependencies-injection) **must** be namespaced.

The convention for namespacing is to prefix your function/constant name with *\<namespace\>\_\<name\>* .

For example, if your namespace is *foo* and you want to write a helper function that returns a pink color, you can write:

```c
// assets/foo/shaders/include/sdk/item.vsh/include.glsl
vec4 foo_getPinkColor() {
    return vec4(1., 0., 1., 1.);
}
```

### SDK inter-stage communication {#sdk-inter-stage-communication}

Permitting everyone to add their own inter-stage variables would be a nightmare for compatibility. Instead, the SDK provides predefined input variables from the vertex to fragment stage. Those variables are:

| Qualifier | Type | Name |
| ----- | ----- | ----- |
| flat | int | sdk\_int |
| flat  | int  | sdk\_int\_b |
| smooth | float | sdk\_float |
| smooth | vec4 | sdk\_vec4 |
| smooth | vec4 | sdk\_vec4\_b |
| flat | ivec4 | sdk\_ivec4 |
| flat | mat4 | sdk\_mat4 |

These variables are already defined, so **do not** redefine them.

As long as your main injection is reached, you are free to read/write/override those variables as you wish. This allows you to transmit data, totalling 88 bytes, from the vertex shader to the fragment shader.

You can of course also override the vanilla outputs in the vertex program.

### Include Guard {#include-guard}

Header guards are common practice in C programming (see [this wikipedia page](https://en.wikipedia.org/wiki/Include_guard)) to avoid multiple inclusions of the same header file. In GLSL, we can achieve a similar effect by using preprocessor directives.

In our context, if you want to import another shader file in your include.glsl, you should wrap the content of the imported file with an include guard to prevent potential conflicts with other namespaces that might import the same file.

For example, if you write \`foo:bar.glsl\` that you’ll want to import later, add guards like this:

```c
// assets/foo/shaders/include/bar.glsl
#ifndef MOJIMPORT_FOO_BAR_GLSL
#define MOJIMPORT_FOO_BAR_GLSL

// write your amazing library here
// /!\ functions written here MUST be namespaced as well, for example 
void foo_bar_myFunction() {
    // some logic
}

#endif // MOJIMPORT_FOO_BAR_GLSL
```

## Identifier

That’s the tricky part of any shader, identifying what and what shouldn’t be treated differently from the vanilla behavior.

With the SDK, any identification goes by using pixel identifiers on the texture. It means that any plane (quad) surface you want to modify **requires** you to remap UV0 manually in the shader if you need the texture.

### Common Identifier pixel

The model UV must be mapped onto a pixel of the following color RGBA(1, 2, 3, 255\) (hex **\#010203FF**).

### Unique identifier pixel

On the immediate right neighbor pixel of the common identifier pixel, must be placed a pixel of the exact color you referenced inside your namespace’s [*sdk.toml*](#sdk.toml-configuration).

### Examples

With that, the surface can be used to be modified in the different shaders.

# **Interactions API**

To aid in booth development, we’ve created a shortcut for running commands from actions on interaction entities. All you need to do is tag the interaction and add some custom data.

| summon interaction \~ \~ \~ {Tags: \["summit.interactable", "summit.static"\], data: {summit\_interactable: {on\_right\_click: "execute on target run function a:bc"}}} |
| :---- |

This will summon an interaction entity that will run function a:bc as the player that right clicked on the interaction. Any command in the on\_right\_click or on\_left\_click fields are run as / at the interaction. To have them target the player you have to include execute on target run for right click or execute on attacker run for left click. 

As a special form of the Interactions system, you can enable a player to sit by right clicking on the interaction. This is done by setting  special: "sit" in the summit\_interactable data field. You can also specify the height above the bottom of the interaction that a player will sit with an optional height field in summit\_interactable, if absent, this will be half of the interaction's main  height field.

| summon interaction \~ \~ \~ {Tags: \["summit.interactable", "summit.static"\], data: {summit\_interactable: {special:"sit",height:0.5}}} Another special form can give players a specific balloon by right clicking on the interaction. # ⭐ Queue API

# **Queue API**

**NOTE**: THIS IS SEPARATE FROM THE BATTLEGROUNDS. DO NOT USE THIS FOR THE BATTLEGROUNDS.

This is an optional system that we encourage booth vendors to hook into if they have an activity they want or need to have players queue for. It involves registering an activity and then a few API functions that you will call.

With this system a player can be queued for one event and if they queue for another will be prompted if they want to switch queues. 

## API Functions

**summit.activity:api/register**

Run to initialize an activity. Activities are stored in markers which are created where this function is run. If there is an existing activity with the same booth\_id and activity\_id, that one will be updated with the provided arguments regardless of location.

Example Usage:

```
function summit.activity:api/register { \
    booth_id: "booth_1", \
    booth_name: "Booth 1", booth_name_color: "red", \
    activity_id: "activity_1", \
    activity_name: "Activity 1", activity_name_color: "#FA3BC0", \
    player_count: 1, \
    start_function: "booth_1:start_activity_1", \
    abort_function: "booth_1:abort_activity_1" \
}
```

Required Macro Arguments: 

- booth\_id \= Should be the same as your booth's namespace  
- booth\_name \= Name of your booth for display  
- booth\_name\_color \= Color to use for the booth\_name. Can be a color name or a hex code  
- activity\_id \= Technical ID for this activity. You don't have to worry about collisions with other booths, this is used in conjunction with the booth\_id  
- activity\_name \= Name of your activity for display  
- activity\_name\_color \= Color to use for the activity\_name. Can be a color name or a hex code  
- player\_count \= Integer for number of players needed for this activity.  
- start\_function \= Function resource location that will be run when there are enough players to start the activity. Players **will not** be teleported. Active players will be tagged with summit.activity.active.\<booth\_id\>.\<activity\_id\> So for the above example this will be summit.activity.active.booth\_1.activity\_1. Use this tag to select players and teleport them to where your activity is.  
- abort\_function \= Function resource location that will be run when the activity is aborted due to all active players being offline for several seconds. Use this to clear the schedule of any ticking functions you may have started from the start\_function and whatever else you may need to do to return the activity to an inactive state. This is necessary, but if you have nothing to clear, just direct this to an empty function.

Returns:  
0 \= No existing marker, creating a new one.  
Anything else, activity was updated with new info and return value represents queue check.

\* causes queue check

### **summit.activity:api/enqueue\_player**

## Queue Check

When a queue check happens, the following things happen:

- The activity's status is checked for if it is active or an error code. Return if applicable.  
- The total number of queued players are checked. Return if not enough players.  
- Those queued players are then checked if they are online. If they are offline, they will be dequeued and prompted to rejoin on reconnect.  
- If there are enough online players queued, the activity will start and the start\_function will be called.  
  - There are plans to prompt players to join the activity when their time in the queue comes, but currently it just throws players in.  
- The function tag \#summit.activity:api/post\_queue\_check/\<booth\_id\>.\<activity\_id\> will be ran after the queue check regardless of how far the queue check got.   
- The caller of the queue check will return the return value from the queue check.

Return Values:  
0 \= Activity Started, start function called  
\-11 \= Activity is already active   
\-12 \= Activity status is an error code (error codes are unused right now)  
\-13 \= Not enough players queued to start activity  
\-14 \= Not enough online players queued

# ⭐ Regen API

# Regen API

All players at Smithed Summit will be given Resistance V and Saturation when their hunger is not full. Vendors who need players to take damage as part of their booth activities will need to give players a specific tag to disable this resistance/regen effect.

Vendors should tag a player who joins their activity with:

**tag @s add summit.no\_regen**

Players with the summit.no\_regen tag will no longer be given the persistent resistance and max saturation effects. Their hunger bar will be depleted and consistently held at a value of 8 (4 hunger bars) so they can sprint, but will not have natural health regen.

Booth vendors should make sure to clear all effects of the player after giving them this tag as to remove the previously applied resistance effect.

**effect clear @s**

Players who are in the battlegrounds (tagged with summit.battlegrounds.player) will also no longer be given the persistent resistance effect. In contrast to players with the summit.no\_regen tag, however, battlegrounds players’ hunger bars will not be manipulated  by the BSL.

Similarly, players with the summit.natural\_regen tag will no longer have persistent resistance AND their hunger bar will not be manipulated at all by summit systems.

# ⭐ Technical Deep-Dive

# **Behind the Scenes**

It’s no secret that fitting hundreds of players on a single Minecraft server will cause performance problems. It’s also even crazier to consider **70+** booths, all containing ticking and custom interactions, running concurrently at scale. In order to achieve these goals for Summit, we have to *heavily* customize how our servers are set up in order to maintain consistent performance throughout Summit.

## Fabric: Our choice of Server Jar

The first choice for any server maintainer is the choice of your server jar. We’ve chosen Fabric as it’s the only ecosystem flexible enough to support our performance and commands needs. Other frameworks such as Paper are notoriously difficult to get running properly with commands and our team is more familiar with the setup that Fabric can provide.

We also can take advantage of the vast open source community to rely on niche tech and compatibility mods. This allows us to reduce our workload immensely and work with others familiar with the requirements that running data packs at scale requires

## In-house Mods

We have created a set of custom mods to help us both during development and actual production Summit server. Outside of [**Unjank**](#unjank) (primarily useful during development), we are working on updating [**Radon**](http://github.com/smithed-mc/radon), which is a general data pack optimization mod, and a custom regional commands mod that will help us split booth commands across multiple threads. This mod also makes big performance gains by reducing the number of entities sent to the client via the summit.static tag (see more [**here**](#static-vs-dynamic-entities)).

Multi-threading is a scary concept as it introduces a lot of potential problems such as race conditions that didn’t exist in normal Minecraft. Since the environment of Summit is quite custom, we’ve designed this mod to make broad assumptions and shortcuts which booth vendors will abide by. For example, each booth has a custom set of function tags to only tick their booth if someone is inside. This lets us ensure that each booth’s logic doesn’t interact across threads letting us avoid potential inconsistencies in behavior.

## Worldguard

Since adventure mode unfortunately allows you to interact with certain blocks that can update the world, we have to implement a worldguard mod that will kill interaction packets sent by the client. Some booths may want to enable block interactivity, which they can do so using a special marker entity with a specific tag. You can read more about that [**here**]().

## Sharding

Those who joined us for Summit ‘24 might remember the server splitting system we had that split our playerbase among different servers. This will be making a grand return where we will be hosting likely \~2-3 copies of the main server (known as shards) that players will connect to automatically (based on each server’s load). Players will be made aware of these via some in-game symbols and will be able to switch freely between them once they join the server. These allow us to “Load Balance” allowing us to handle much higher player counts as we will be leveraging more of the server hardware.

We have learned a lot from our sharding of the first Summit and will be keeping a much closer eye on system metrics to avoid each shard from stealing resources from each other, capping performance.

### Velocity

This is a common and popular proxy framework that lets us have a singular IP address for Summit while transferring and handling people across our shards. This also lets us share chat and other custom features across our shards.

### Player Ghosting (TBD)

This is a new system we are investigating that will allow us to simulate the experience of one large server by showing nearby players from other shards to each player. Ghosted players will have a special indicator of what shard they actually live on and you’ll be able to see them move, see what items and armor they are wearing, and chat with them in-game. You will *not* be able to interact with them or see them engage with specific world interactions (such as the battlegrounds, minigames, etc). If you wish to interact with a friend on a different shard, you’ll have to switch shards via our in-game menus (TBD).

⚠️​This system is heavily in the research phase and may be completely scrapped if it’s not feasible.

### Shard Syncing {#shard-syncing}

Since we will be hosting multiple copies of each server, you may be wondering what happens to a player’s data and progress as they switch between these servers. We employ a custom syncing solution known as [**Summit Sync**](https://github.com/Smithed-MC/summit-sync). We use this to specifically sync advancements, which we then built on top of for things like the Sticker Book and Plushie progress.

But how should *you* sync the data you care about?  
Ping `@TheNuclearNexus` if you have questions.

- **Advancement**  
  - If all you care about is whether someone unlocks something, use an advancement. Advancements are automatically synced between shards.  
  - We can organize with you after the event starts to get the number of users who unlocked your advancement, if that is something you need  
- **Items**  
  - The user’s inventory and ender chest are synced. Use this when you want to store item-specific information persistently. Good for long cooldowns, item states, etc.  
- **Scores**  
  - Using a `SyncSettings` file (more information in the section below), you can define scores that are synced when a player leaves/joins a shard. These do **NOT** update live between shards. Only player scores will be synced.  
- **PlayerDB**  
  - This solution should be used sparingly, but it is the most powerful way to sync data across shards. Similar to scores, you can configure this in a `SyncSettings`; you’ll index the data based on the user’s UUID, modify it, and then signal to the server that the data has changed.   
  - Do not modify these tickly, at best every 5 ticks, ideally a second or longer.   
  - If you need to iterate over the list, there is a `keys` field that contains a list of `{uuid: “<uuid>”}` entries for each key in your database.

#### Example database modification

```kotlin
data modify storage test:db database.$(uuid).foo set value 1
data modify storage test:db dirty.$(uuid) set value 1b
```

#### Example `on_sync` handler

```kotlin
//#> `on_sync` is always provided with the user's uuid as a macro argument
$tellraw @a [                                              \
    "A user was synced with foo '",                        \
    {"nbt": "database.$(uuid).foo", "storage": "test:db"}, \
    "'"                                                    \
]
```

#### Example `SyncSettings` (‘data/\<namespace\>/sync\_setting/\<name\>.json’)

```json
{
  "scores": ["foo.bar"], // List of objectives
  "databases": [
    {
      "key": "test:db",                // The storage to use for the database
      "on_sync": "test:events/on_sync" // The function to run when data is synced 
                                       // to the database
    }
  ]
}
```

# ⁉️ **FAQ**

# ***Does my booth need to be manned during Summit?***

No\! Summit booths are mostly unmanned and most booth vendors are traversing the rest of the server. You can *optionally* hang out at your booth if you want but we also recommend registering for an event and hosting a Meet & Greet for something a bit more purposeful.

# ***What Minecraft version is being used for Summit?***

We have updated to 26.2 so be sure to update your personal worlds and packs to ensure you are compatible.

Outdated:  
*Currently, we are targeting Minecraft 26.1. If 26.2 releases before our booth freeze, we may consider updating to it but it’s dependent on whether it breaks too much and how close it is to the booth freeze. Do not expect or rely on new features in your booth’s design, we’ll make an announcement closer to the date as we learn more.*

# ***Can I use hardcoded UUIDs for my entities?***

Yes, but you must generate a new one and use that. We do not allow for custom hard coded UUIDs due to the greater risk of collision. Here’s a link to help with that process   
[**https://www.soltoder.com/mc-uuid-converter/**](https://www.soltoder.com/mc-uuid-converter/)

# ***Can I use custom shaders / vanilla shaders / objmc?***

Only shaders using the **[Shader Development Kit]()** will be supported. Unfortunately other custom shaders are tough to make consistent when operating with multiple packs.

# ***Can I use Animated Java?***

Yes\! We are actively using AJ throughout the build server. AJ even has a booth\!  
See our guide on it [**here**]().

# ***Can I override vanilla block states?***

No. We’re already overriding many for the build of the world itself and there are simply not enough for us to allow vendors to do that as well. Use item displays.

# ***Will blocks be interactable on the live server?***

We will have a system for this a little later down the line. The default will be that players will not be able to interact with blocks and we'll whitelist specific ones. Keep this in mind for when we have that system fully set up.

# ***Can I use command blocks?***

If you really need to, sure. However, it is much preferred to use a data pack. We will need to review any commands you use and command blocks make that process more difficult.

# ***What is Unjank and will it be on the Summit server?***

Unjank is a helpful mod developed by us that adds a couple of QOL features related to data packing, you can read about it [**here**](#unjank).

On the production server, the primary part of the mod that’ll be functioning will be the removal of chat spam due to dialog triggers (only a problem on servers as it can kick you so not relevant for single player worlds.)

# ***Can I use the scoreboard sidebar?***

Unfortunately, scoreboard sidebars are still global (and limited to 16 team colors). There’s no good way for us to coordinate the use of these so we will not be permitting their usage. There potentially will be some usage available in the baobab battlegrounds, more info to come.

# ***Can I use advancements?***

It matters\! If they are technical, non-visual advancements, yes, we encourage them for performance. If they are normal (visual) advancements used for tracking player progress, then you should not – we’ve developed an alternative system for player achievements called the [**Sticker Book**]()\!

If you want to show off advancements as a part of your booth experience, you should make sure to only grant them when a player is in your booth and clean them up / revoke them when they leave.

# ***Can I have mobs in my booth?***

NoAI mobs are ok.  
For mobs with AI, they must meet the following conditions:

- They must not be able to wander out under any circumstances  
- They must not be destructive and they must not fire projectiles outside of your booth  
- Obviously tag them the same as any other entity  
- It might be a good idea to remove their AI if there is no player in your booth

# ***Can I use custom maps in my booth?***

For most booths, no. Please leverage item\_models as our preferred method of getting custom images in-game. If you feel like you need to use custom maps for a specific reason, please reach out to your coordinator, and we can find a solution that works for you.

# ***Can I use the xp bar?***

Yes. We are currently deciding internally if we would like to have vendors store and restore the xp of a player or to just set the xp level back to 0 when you are done.

# ***Can I display information using a \`below\_name\` scoreboard***

You can use the scoreboard \`summit.below\_name\` to display scores under the name tags of entities. If you use this scoreboard, you should reset the score of players when it is not used and when they exit your booth.  The display name of \`summit.below\_name\` is empty, so the score is displayed as "\<number\> ", it can be used in association with \`/scoreboard players display\`..

# ***Birds are loud and annoying\! How do I silence them when people are in my booth?***

Players who have the summit.silence\_birds tag won’t be able to hear any sounds birds make. Add that tag to players who enter your booth, but be sure to remove it when they leave\!
